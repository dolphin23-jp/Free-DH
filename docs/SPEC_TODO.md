## COMBAT_SPEC §2: `fork(seed, label)` のハッシュ方式

- **箇所**: `docs/COMBAT_SPEC.md` §2「fork = `hash(親シード文字列 + ":" + ラベル)`」
- **問題**: `hash` のアルゴリズム、文字エンコーディング、出力ビット幅が指定されていない。方式が変わると敵順・ドロップ・ショップ・戦闘内乱数の全固定シード結果が変わる。
- **暫定判断**: `${String(seed)}:${label}` に対する32-bit FNV-1aを使用する。文字列はJavaScriptのUTF-16コード単位として走査し、結果をunsigned 32-bit整数としてmulberry32へ渡す。
- **理由**: 外部依存がなく、ブラウザとNodeで同一結果になり、実装と固定ベクトルが小さいため。
- **今後の確認**: セーブ互換性やゴールデンテストを固定する前に、この方式を正式仕様として採用するか確認する。変更する場合はT02の固定ベクトルと後続の全ゴールデンテストを更新する。

## COMBAT_SPEC §5/§12: `damageReduction` の適用位置

- **箇所**: §5のダメージパイプラインには `damageReduction` が現れず、§12のパッシブ語彙にのみ定義されている。
- **問題**: 加算・乗算・会心・特効・端数処理・ブロックのどの段階で固定値軽減を差し引くかが未指定。適用位置により会心時や小数ダメージの結果が変わる。
- **暫定判断**: 特効倍率の適用後、端数処理の直前に固定値を減算し、0未満にはしない。その後、小数第1位に丸めてブロックまたはHPへ適用する。
- **理由**: 固定値軽減をブロックとは独立した被ダメージ軽減として扱え、会心・特効後の最終威力から一定値を引く挙動になるため。
- **今後の確認**: 正式な適用位置が決まった場合、`calculateDamage` とT04の固定テストを同時に更新する。

## COMBAT_SPEC §6: 火傷バッチの満了tick

- **箇所**: 火傷は「付与ごとに独立バッチ」「各3.0秒で消滅」、状態異常ダメージは整数秒に処理するとある。
- **問題**: 付与からちょうど3.0秒後が整数秒と一致した場合、そのtickのダメージを適用してから消滅するか、適用前に消滅するかが未指定。
- **暫定判断**: 整数秒の状態異常ダメージを先に処理し、そのtickのCD進行後に残り時間を減らす。新規付与は同tickでは時間を消費せず、各バッチは最大3回の整数秒ダメージを発生させる。
- **理由**: 付与時刻によって2回または3回へ揺れず、3秒バッチの期待ダメージを決定的に保てるため。
- **今後の確認**: 満了tickをダメージ前に消滅させる仕様へ変更する場合、火傷の期待値とT05の固定テストを更新する。

## COMBAT_SPEC §6: 同一実体の毒・火傷の処理順

- **箇所**: tick内ではPlayer→Enemyの順序が指定されているが、同一実体に毒と火傷が同時にある場合の順序は未指定。
- **問題**: 片方のダメージでHPが0になった場合、死亡判定が即時なのでイベントログと後続ダメージの有無が変わる。
- **暫定判断**: 同一実体では毒を先、火傷を後に処理し、各適用直後に終了判定する。
- **理由**: 固定順を設けてイベントログを決定論的にし、毒スタックを単一値、火傷を複数バッチとして扱う状態構造とも整合するため。
- **今後の確認**: 正式順序が決まった場合、状態異常イベントのゴールデンテストを更新する。

## COMBAT_SPEC §7: `hpBelow` の閾値データ表現

- **箇所**: §7は `hpBelow(閾値を下回った瞬間)` と定義するが、元のitems.jsonトリガー語彙には閾値フィールドがない。
- **問題**: 効果量の `value` とHP閾値を同時に表現できず、任意閾値のトリガーをデータから構築できない。
- **暫定判断**: トリガースキーマと解決済みトリガー入力に任意の `thresholdPercent` を追加し、省略時は50%とする。既存JSONの数値・レコードは変更しない。発火済みキーを戦闘状態へ保存し、同じトリガーは1戦闘に1回だけ発火する。
- **理由**: JSON互換性を保ったまま任意閾値を型・スキーマで検証でき、将来データにhpBelowが追加されても同じ経路を使えるため。
- **今後の確認**: `thresholdPercent` と省略時50%を正式仕様として採用するか、T08前に決定する。

## COMBAT_SPEC §7: 命中後トリガーの細部順序

- **箇所**: §5(g)は `onHit` の後に被弾側 `onDamaged` とするが、`onBlocked` と `hpBelow` の相対位置は未指定。
- **問題**: 反射・回復・閾値回復が同じ一撃で成立した場合、最終HPや決着の有無が順序で変わる。
- **暫定判断**: `onHit → onDamaged → onBlocked → hpBelow` の順とする。`onDamaged` はHPまたはブロックが1以上減った場合、`onBlocked` は全額をブロックした場合のみ発火する。
- **理由**: 仕様に明記されたonHit→onDamagedを維持し、完全防御の追加処理とHP閾値判定をその後に一意化できるため。
- **今後の確認**: 正式順序が決まった場合、T06のトリガーイベント列テストを更新する。

## COMBAT_SPEC §7: トリガー由来ダメージの会心・ブロック規則

- **箇所**: 連鎖深度1は規定されているが、`damage`/`reflect`トリガーが会心判定やブロックを通るかは未指定。
- **問題**: 通常ダメージパイプラインを全て通すと乱数消費順が増え、反射をTrueダメージにすると防御価値が変わる。
- **暫定判断**: トリガー由来ダメージは非会心の通常ダメージとしてブロック可能にする。乱数は消費せず、`triggersAllowed: false`として新たなonHit/onDamaged/onBlocked/hpBelowを発生させない。
- **理由**: 反射を防御可能に保ちつつ、連鎖深度1と戦闘内乱数の限定条件を確実に守れるため。
- **今後の確認**: 反射・トリガーダメージを別種別にする場合、ダメージ記録とゴールデンテストを更新する。

## COMBAT_SPEC §7: `onKill` の発火範囲

- **箇所**: `onKill` は定義されているが、撃破したアイテム自身だけか、カバン内の全onKillトリガーかが未指定。
- **問題**: 毒・火傷・反射による撃破では単一の攻撃アイテムへ帰属できない場合がある。
- **暫定判断**: 勝者側の未封印実体が持つonKillをグリッド順で全て1回発火し、その後battleWinを同順で解決する。
- **理由**: 状態異常や反射撃破でも一貫して処理でき、アフィックスの収奪効果を取りこぼさないため。
- **今後の確認**: 攻撃元限定へ変更する場合、状態異常・反射撃破時の帰属規則を併せて定義する。
